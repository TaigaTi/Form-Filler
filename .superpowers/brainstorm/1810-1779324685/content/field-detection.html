Field Detection & Data Generation

How labels are found and how values are chosen

Label Detection (priority order)

For each input/select/textarea, the first source that yields text wins:

① aria-label / aria-labelledby
Explicit accessibility labels — highest reliability
② <label for="id"> association
Standard HTML label linkage via for/id
③ Wrapping <label> parent
Input nested inside its own label element
④ Preceding sibling text / <p> / <span>
Non-standard label placed before the field in DOM
⑤ placeholder attribute
Fallback when no visible label exists
⑥ name / id attribute
Last resort — often camelCase/snake_case, normalized

Data Generation Pipeline

STEP 1 — FIELD TYPE HANDLING
select → pick 2nd non-blank option
checkbox → check it
radio group → select first option
date → random past date
number → random within min/max
file → skip (can't fill programmatically)
STEP 2 — RULES ENGINE (local, instant)
Normalized label matched against ~40 patterns:
"email" → faker.internet.email()
"first name" → faker.person.firstName()
"phone / mobile / tel" → faker.phone.number()
"address / street" → faker.location.streetAddress()
"message / comment / note" → faker.lorem.paragraph()
"password" → "TestPassword123!"
+ ~35 more patterns
STEP 3 — CLAUDE HAIKU FALLBACK (unmatched fields only)
Batches all unmatched labels into one API call:
Fill these form fields with realistic
fake data. Return JSON only.

Fields: ["testimonial",
  "reason for selling goods"]
Response cached in chrome.storage.local by label text to avoid repeat API calls.
React / Vue compatibility
After setting element.value, the content script dispatches synthetic input and change events so framework-managed fields (React controlled inputs, Vue v-model) properly update their internal state.
